# specd workflow feedback log

Append-only log of friction hit while *using* specd to build aido. This repo is
not where specd is fixed — entries here flow upstream as text.

Do not delete entries. Do not rewrite history — resolved entries get
`Status: resolved (<upstream commit/spec>)`, they stay.

Two entry kinds: **friction** (something blocked or misled the agent) and
**improvement** (workflow succeeded, but a concrete change would make it
faster, clearer, or harder to get wrong).

## Friction format

```markdown
### <YYYY-MM-DD> — friction — <short title>
- **Context:** spec slug, phase, role/mission, exact command run
- **Expected:** what the agent believed would happen and why
- **Actual:** exact output / exit code / blocker (shortest decisive line)
- **Root cause:** harness bug | missing guidance | ambiguous docs | agent error
- **Recommendation:** concrete change (verb/flag/gate/doc/message text)
- **Status:** open
```

## Improvement format

```markdown
### <YYYY-MM-DD> — improvement — <short title>
- **Context:** spec slug, phase, role/mission, command sequence that worked
- **Observation:** what cost time or attention despite succeeding
- **Cost:** turns, re-reads, redundant commands, or context tokens burned
- **Recommendation:** concrete change (verb/flag/gate/doc/message text)
- **Tradeoff:** what it costs — invariant risk, added surface, or none
- **Status:** open
```

Improvement entries must clear a bar: an agent-hours or correctness win a
future run can feel. "Nice to have" is not an entry. Anything that would weaken
determinism, evidence integrity, or add a bypass gets logged with the tradeoff
stated plainly and stays a proposal — never a self-applied change.

### 2026-07-21 — friction — steering managed region forces placeholder text ahead of real project law
- **Context:** no spec slug (steering authoring, pre-spec). Editing
  `.specd/steering/{product,tech,structure,workflow,reasoning}.md` after
  `specd init`. No command failed; the blocker is structural.
- **Expected:** steering files are project law, so filling them in means
  replacing the scaffold prose — `"## Thesis"` / `"- **What this product is:** <one sentence>"`
  in `product.md`, `"- **Language / runtime:** <e.g. Go 1.22, stdlib only>"` in
  `tech.md`, `"- **<dir>/** — <what lives here>"` in `structure.md`.
- **Actual:** every one of those lines sits inside
  `<!-- specd:managed:steering/product.md:v1 begin -->` … `end`, guarded by
  `"> Regenerated by `specd init --repair/--refresh`; edit below the closing marker or your changes are lost."`
  The `specd-context` block that drives manifest selection is inside the same
  managed region, so the region cannot be edited at all without risking both the
  content and the file's presence in the manifest. Result: the only durable
  place for real law is *below* the closing marker, and every future context
  manifest now ships unfilled templates (`<one sentence>`, `<what lives here>`)
  as the first thing an agent reads, with the actual binding rules after them.
- **Root cause:** harness design — the managed region mixes machine-owned
  metadata (`specd-context`) with human-owned prompt prose in one
  clobber-on-refresh block.
- **Recommendation:** split the region. Keep `specd-context` (plus the marker
  pair) machine-owned and regenerable; move the `## Thesis` / `## Stack` /
  `## Layout` prompt bodies outside it, so `--repair` restores metadata without
  reasserting placeholders. Failing that, have `init --repair/--refresh` drop
  the prompt prose once any content exists below the closing marker.
- **Status:** open

### 2026-07-21 — improvement — no way to ask whether steering is actually filled in
- **Context:** no spec slug. After authoring all five steering files, wanted to
  confirm they parse, are selected by the manifest, and carry real content. Ran
  `specd --help` to look for a verb.
- **Observation:** `specd check <slug>` validates a spec, and `specd context <slug> <task> --json`
  needs a slug and a task — both require a spec to exist. During bootstrap
  (steering written before the first `specd new`) there is no slug, so there is
  no command that answers "is my steering valid and will it load?". Verification
  fell back to `git diff --numstat` to prove the managed regions were byte-identical.
- **Cost:** one extra `--help` round trip plus a manual diff check, and no
  positive confirmation at the end — only evidence that nothing was deleted.
- **Recommendation:** add `specd steering [--json]`: list each steering file with
  its `specd-context` id/version/priority, managed-region digest match, byte
  size below the closing marker, and a `filled|placeholder` flag derived from the
  scaffold text still being present. Read-only, no slug required.
- **Tradeoff:** new read-only surface only; no gate, evidence, or determinism
  impact.
- **Status:** open
